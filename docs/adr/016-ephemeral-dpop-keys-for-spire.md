# ADR-016: Ephemeral DPoP Keys for SPIRE

## Status

Accepted

## Context

For SPIRE-backed authentication, prmana must decide whether DPoP proofs are signed
with the workload's SVID private key or with a separate ephemeral key pair generated by
the local signer.

Two approaches were considered:

- Sign DPoP proofs with the JWT-SVID's associated private key
- Generate an independent ephemeral P-256 key pair for DPoP and use the JWT-SVID only
  as the access token

Using the SVID private key would tightly couple session proof-of-possession to a
longer-lived workload identity and assumes that key material is always accessible to
the agent. RFC 9449 DPoP only requires proof-of-possession for the presented token; it
does not require reuse of the token issuer's signing key.

## Decision

The accepted design for `SpireSigner` is to generate independent ephemeral P-256 key
pairs for DPoP proofs. The JWT-SVID is used strictly as the access token. DPoP proofs
are not signed with the SVID private key.

## Consequences

### Positive

- Better isolation between workload identity material and per-session DPoP keys
- Works in environments where the SPIRE-managed private key is not directly exposed
- Limits the blast radius of DPoP key compromise to a session attempt
- Aligns cleanly with RFC 9449 proof-of-possession semantics

### Negative

- Adds another key lifecycle to manage in the agent
- DPoP key continuity is not tied to the workload's long-lived SPIFFE identity
- Some operators may initially expect SVID-key reuse and need documentation
- Requires dedicated agent-side implementation work for `SpireSigner`
