**Title:** `# Identity Rationalization Guide`

**Subtitle:** `Deploying unix-oidc alongside FreeIPA and Azure Entra ID`

**Target audience note:** Enterprise identity architects and Linux admins managing mixed AD/FreeIPA/cloud identity environments.

**Table of Contents sections (8 sections):**

1. **The Problem: Identity Archaeology**
   - The "archaeology problem": when someone leaves, how do you find all their SSH access? authorized_keys files scattered across servers, no central audit trail.
   - unix-oidc solves this: single identity source (IdP), audit log with username + issuer + session, centralized revocation at IdP.
   - Diagram: Traditional (keys everywhere) vs unix-oidc (central IdP + SSSD).

2. **Design Anchor: SSSD is the Source of Truth for Groups**
   - Group membership is resolved from SSSD/NSS, NOT from OIDC token claims.
   - Why: FreeIPA is the authoritative source for Unix group membership. Token claims reflect IdP group assignments which may not match Unix GIDs.
   - Design decision (Phase 8): `GroupSource::TokenClaim` is not supported — it was removed as dead code in Phase 26. The only supported group resolution path is SSSD/NSS.
   - What this means: groups in `login_groups` and `sudo_groups` config must be Unix group names present in SSSD/NSS, not IdP claim values.
   - Implementation: `pam-unix-oidc/src/policy/config.rs` GroupMappingConfig.

3. **UPN-to-UID Mapping: Worked Examples**

   3.1 FreeIPA native users (no mapping needed):
   - FreeIPA user `alice` has `preferred_username: alice` → maps directly, no transform.
   - Config: no `claim_mapping` section needed.

   3.2 Entra ID UPN mapping (strip_domain):
   - Entra user `alice@corp.example` has `preferred_username: alice@corp.example` (UPN format).
   - To map to local user `alice`: configure `strip_domain: true` on the Entra issuer.
   - Full config example:
     ```yaml
     issuers:
       - issuer_url: "https://login.microsoftonline.com/{TENANT_ID}/v2.0"
         client_id: "your-client-id"
         dpop_enforcement: disabled   # Entra doesn't support DPoP
         claim_mapping:
           username_claim: preferred_username
           strip_domain: true          # alice@corp.example → alice
     ```
   - Implementation: `pam-unix-oidc/src/oidc/username.rs` — `strip_domain_transform()`.
   - Warning: if two Entra users have the same username prefix (alice@corp.example and alice@other.example), the second will collide. unix-oidc detects this at config load time and hard-fails. This is by design (IDN-03).

   3.3 Custom claim mapping (email as username):
   - Some IdPs use `email` instead of `preferred_username`.
   - Config: `username_claim: email` with `strip_domain: true`.

   3.4 Static username map:
   - For legacy systems where IdP username doesn't match local username at all.
   - Config example with explicit map.

   3.5 Identity collision detection:
   - Explain how unix-oidc prevents two IdP identities mapping to the same local user (hard-fail at startup, not at auth time).

4. **FreeIPA + Entra Coexistence Patterns**

   4.1 Pattern A: FreeIPA-only (single issuer, Keycloak federating to FreeIPA)
   - Architecture: FreeIPA ↔ Keycloak (via LDAP federation) ↔ unix-oidc
   - Users exist in FreeIPA; Keycloak issues tokens with FreeIPA UIDs as `preferred_username`.
   - Group membership from FreeIPA via SSSD.
   - Config example: single issuer, no strip_domain.
   - Use case: greenfield, no Microsoft dependency.

   4.2 Pattern B: FreeIPA + Entra (dual-issuer)
   - Architecture: FreeIPA users + Entra users, two `issuers[]` entries in policy.yaml.
   - FreeIPA users authenticate via Keycloak (first issuer). Entra users authenticate directly (second issuer, strip_domain=true).
   - Group membership for both: from SSSD/NSS (not from either IdP's token claims).
   - Critical: FreeIPA and Entra UIDs must not collide. Use prefixes or explicit maps if needed.
   - Config example: full two-issuer policy.yaml.
   - Use case: migration from Entra-only to FreeIPA, or permanent coexistence.

   4.3 Pattern C: Entra-only (no FreeIPA)
   - Architecture: Entra + SSSD joined to Entra Domain Services (or manual /etc/passwd).
   - Users exist in Entra; SSSD syncs via Azure AD (SSSD >= 2.5 supports it).
   - Group membership from SSSD (Azure AD groups synced to local groups).
   - Config: single Entra issuer, strip_domain=true, dpop_enforcement=disabled.
   - Use case: Azure-native enterprises.

5. **Group Sync Strategies**

   5.1 IPA-AD Trust (FreeIPA + Active Directory):
   - FreeIPA establishes a cross-forest trust with Active Directory.
   - AD users can log in using AD credentials; groups from AD appear in FreeIPA/SSSD.
   - Pros: transparent to unix-oidc; group membership correct from day 1.
   - Cons: requires domain-level trust setup; complex in multi-forest environments.
   - When to use: enterprise with established AD + FreeIPA already in use.

   5.2 SSSD Multi-Domain:
   - SSSD configured with multiple `[domain/...]` sections: one for FreeIPA, one for Entra/AD.
   - Groups from each domain resolve locally via NSS.
   - Pros: no cross-forest trust needed; works with Entra Domain Services.
   - Cons: group name conflicts possible if both domains have group `admins`.
   - When to use: migration period, parallel identity systems.

   5.3 Manual Group Sync:
   - A cron job or CI pipeline syncs group membership from IdP to /etc/group or LDAP.
   - Pros: simple, works anywhere.
   - Cons: lag between IdP group change and local group update.
   - When to use: small deployments, testing.

   5.4 Recommendation matrix: table of use cases → recommended strategy.

6. **The Archaeology Problem: Access Audit on Offboarding**

   6.1 Finding all access for a departing user:
   - Traditional problem: grep authorized_keys on every server.
   - With unix-oidc: query audit logs for `preferred_username = alice` across all servers.
   - Example jq query against JSON audit log.
   - Example grep query against syslog.

   6.2 Revoking access:
   - Step 1: Disable account in IdP (Entra/Keycloak). Active tokens expire within their lifetime (default ≤ 1 hour).
   - Step 2: If immediate revocation needed: configure `token_introspection` to poll IdP. Revocation takes effect within introspection cache TTL (default 60s).
   - Step 3: Remove from FreeIPA/SSSD groups. Active sessions using group-based access are not affected until next auth — document this limitation.

   6.3 Audit log query examples:
   - "Show all auth events for alice in the last 30 days"
   - "Show all sudo step-up events for alice"
   - "Show sessions still open for alice after offboarding"

7. **Troubleshooting**

   Table: common failure modes specific to multi-IdP identity setups:
   | Symptom | Cause | Resolution |
   |---------|-------|------------|
   | "User not found" after OIDC success | preferred_username from IdP doesn't match local Unix user | Check strip_domain config; verify SSSD sync |
   | Identity collision at startup | Two Entra UPNs map to same local user | Add explicit static_map or use prefix transform |
   | Group policy denial for Entra user | Login group exists in Entra but not in SSSD/NSS | Sync Entra group to local group via SSSD or manual |
   | "Unknown issuer" for Entra tokens | Tenant ID in issuer URL doesn't match config | Verify issuer_url format: https://login.microsoftonline.com/{TENANT_ID}/v2.0 |
   | DPoP validation error on Entra tokens | Entra doesn't support DPoP | Set dpop_enforcement: disabled on Entra issuer |
   | ACR mismatch | Entra returns different ACR values than Keycloak | Use acr_mapping to normalize; or don't configure required_acr for Entra issuer |

8. **Reference: Configuration Snippets**

   Full working policy.yaml examples for each coexistence pattern from Section 4.
