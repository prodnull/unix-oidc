  **SBUG-01 — Fix oidc_issuer in token_validation_failed calls:**

  In `lib.rs`, the `authenticate()` error handler (lines 420-482) passes `oidc_issuer: None` to every `token_validation_failed()` call. The issuer IS available:
  - For `AuthError::UnknownIssuer(iss)` — use `Some(iss.as_str())`
  - For all other token/DPoP errors — the issuer must be extracted from the token before the error branch. Before the match block (around line 340 where `authenticate_with_multi_issuer` is called), capture the issuer from the successful token parse. If the token was parseable, the issuer is in the claims. If not, pass None (genuine unknown).

  Approach: After the token is extracted from PAM conversation (around line 290), do a best-effort issuer extraction using `extract_iss_for_routing()` (already exists in auth.rs). Store as `let token_issuer: Option<String>`. Thread this into the error match arms as `token_issuer.as_deref()`.

  **SBUG-02 — Wire break_glass.alert_on_use:**

  1. In `audit.rs`, add an `alert_on_use: bool` field to `BreakGlassAuth` variant. Update `break_glass_auth()` constructor to accept the flag. Update `syslog_severity()` to return `Critical` when `alert_on_use=true`, `Info` when `false`. The serialized JSON should include `"severity": "CRITICAL"` or `"severity": "INFO"` accordingly.

  2. In `lib.rs` line 129, change `AuditEvent::break_glass_auth(&pam_user, source_ip)` to `AuditEvent::break_glass_auth(&pam_user, source_ip, policy.break_glass.alert_on_use)`.

  3. Update ALL existing tests in audit.rs that call `break_glass_auth` to pass `true` (preserving current behavior in tests).

<task type="auto" tdd="true">
  <name>Task 2: Fix SBUG-03 (preferred_username=None graceful handling)</name>
  <files>pam-unix-oidc/src/auth.rs, pam-unix-oidc/src/sudo.rs</files>
  <behavior>
    - Test: authenticate_with_multi_issuer with a token missing preferred_username but having email claim configured as username_claim succeeds
    - Test: authenticate_with_multi_issuer with a token missing preferred_username and username_claim="preferred_username" returns IdentityMapping error (not panic)
    - Test: sudo step-up with a token missing preferred_username returns UserMismatch or appropriate error (not panic)
  </behavior>
  <action>
  **auth.rs line 207:**
  The `raw_claim` variable is used only for the `mapped_from` audit field. When `preferred_username` is None and the mapper uses a different claim (e.g. `email`), the `mapped_from` should reflect what the mapper actually used.

  Change:
  ```rust
  let raw_claim = claims.preferred_username.clone().unwrap_or_default();
  ```
  To:
  ```rust
  let raw_claim = claims.get_claim_str(&issuer_config.claim_mapping.username_claim)
      .unwrap_or_default();
  ```
  This extracts the ACTUAL claim the mapper will use, not always preferred_username. If the configured claim is missing, `raw_claim` is empty, which is fine — the mapper's `MissingClaim` error will fire shortly after on line 209.

  **sudo.rs line 516:**
  The `token_user` is used for username mismatch validation. When `preferred_username` is None, comparing "" to the sudo user will always produce UserMismatch, which is correct behavior but produces a confusing error message ("token user '' does not match sudo user 'alice'").

  Change:
  ```rust
  let token_user = claims.preferred_username.clone().unwrap_or_default();
  if token_user != ctx.user {
  ```
  To:
  ```rust
  let token_user = claims.preferred_username.as_deref().or(Some(&claims.sub));
  let token_user_str = token_user.unwrap_or(&claims.sub);
  if token_user_str != ctx.user {
      return Err(SudoError::UserMismatch {
          token_user: token_user_str.to_string(),
          sudo_user: ctx.user.clone(),
      });
  ```
  This falls back to `sub` claim (which is always present per OIDC Core) when `preferred_username` is absent, matching what many IdPs use as the user identifier. The error message will show the actual sub value instead of empty string.
  </action>

  </verify>

    - auth.rs uses the configured username_claim for the raw_claim audit trail field
    - sudo.rs falls back to sub claim when preferred_username is absent
    - No panics or unwrap crashes on tokens missing preferred_username
  </done>
