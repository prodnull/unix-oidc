    - Test 2: A degraded issuer with elapsed recovery interval returns `is_degraded() == false` (should retry)
    - Test 3: A successful JWKS fetch clears the degraded state (failure count reset to 0)
    - Test 4 (negative): A single JWKS failure does NOT mark the issuer as degraded
    - Test 5 (negative): A degraded issuer within its recovery interval remains degraded (no premature retry)
    - Test 6: Health state persistence: write state to temp file, read it back, values match
    - Test 7: Corrupt health file is handled gracefully (treated as healthy, WARN logged)

    MIDP-11 config hot-reload:
    - Test 8: Config freshness check detects changed mtime and reloads
    - Test 9: Config freshness check with unchanged mtime returns cached config
    - Test 10 (negative): A bad YAML file on reload keeps the previous valid config
    - Test 11 (negative): A missing config file on reload keeps the previous valid config
  </behavior>
  <action>
    **MIDP-10: Health monitoring**

    In `pam-unix-oidc/src/policy/config.rs`:
    1. Add `recovery_interval_secs` field to `IssuerConfig` with `#[serde(default = "default_recovery_interval")]` (default 300s).
    2. Create `IssuerHealthState` struct: `{ failure_count: u32, last_failure: Option<i64>, degraded: bool }` with `Serialize`/`Deserialize`.
    3. Create `IssuerHealthManager` with methods:
       - `load(issuer_url: &str) -> IssuerHealthState` — reads from `/run/unix-oidc/issuer-health/{sha256-prefix}.json`, returns default (healthy) on any error
       - `save(issuer_url: &str, state: &IssuerHealthState)` — atomic write (tmp + rename)
       - `record_failure(issuer_url: &str)` — increment failure count, set last_failure, mark degraded if count >= 3
       - `record_success(issuer_url: &str)` — reset failure count, clear degraded flag
       - `is_degraded(issuer_url: &str, recovery_interval_secs: u64) -> bool` — true if degraded AND within recovery interval
    4. The health directory path should be configurable via `UNIX_OIDC_HEALTH_DIR` env var for testing, defaulting to `/run/unix-oidc/issuer-health/`.
    5. File naming: SHA-256 of issuer URL, first 16 hex chars + `.json` to avoid path issues with URL characters.

    In `pam-unix-oidc/src/auth.rs`:
    1. Before calling `jwks_registry.get_or_init()`, check `IssuerHealthManager::is_degraded()`. If degraded, emit `tracing::warn!` with issuer URL and skip this issuer. For multi-issuer, this means falling through to the next candidate. For the case where ALL issuers are degraded, return `AuthError::Config("All configured issuers are degraded")`.
    2. After a successful JWKS fetch (no error from validation), call `record_success()`.
    3. After a JWKS fetch failure, call `record_failure()`.
    4. When an issuer transitions to degraded (failure_count crosses 3), emit an `ISSUER_DEGRADED` audit event via the existing `AuditEvent` enum. When it recovers, emit `ISSUER_RECOVERED`.
    5. Add new `AuditEvent` variants: `IssuerDegraded { timestamp, issuer_url, failure_count, host }` and `IssuerRecovered { timestamp, issuer_url, host }` with `syslog_severity()` of Warning and Info respectively.

    **MIDP-11: Config hot-reload**

    In `pam-unix-oidc/src/policy/config.rs`:
    1. Create `ConfigCache` struct with `{ config: PolicyConfig, mtime: SystemTime, path: PathBuf }`.
    2. Create a `static CONFIG_CACHE: Lazy<Mutex<Option<ConfigCache>>>` for cross-call persistence.
    3. Create `PolicyConfig::load_fresh()` method: stat the config file, compare mtime to cached, re-parse if changed. On parse error, log WARNING and return the cached config. On success, update cache and return new config.
    4. The config path is resolved from `UNIX_OIDC_POLICY` env var or `/etc/unix-oidc/policy.yaml` default (same as `from_env()`).

    In `pam-unix-oidc/src/lib.rs`:
    1. Replace `PolicyConfig::from_env().unwrap_or_default()` in `pam_sm_authenticate` with `PolicyConfig::load_fresh()` for the multi-issuer path. The legacy single-issuer path can continue using `from_env()`.

    **Important constraints:**
    - Health state is file-based because each forked sshd process is ephemeral (no shared memory). All state must go through filesystem.
    - Config reload uses stat() not inotify — simpler, works on all platforms, no file descriptor management.
    - All file I/O is best-effort: failures are logged at WARN and never block authentication.
