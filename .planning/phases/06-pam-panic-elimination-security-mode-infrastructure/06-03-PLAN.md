  <name>Task 1: Wire enforcement modes into validation and auth pipeline</name>

    - Test: With jti_enforcement=warn and a token missing JTI -> validation succeeds (with tracing warn)
    - Test: With jti_enforcement=disabled and a token missing JTI -> validation succeeds (no log)
    - Test: With dpop_required=strict -> existing DPoP-required behavior unchanged
    - Test: Default SecurityModes (v1.0 compat) produces same behavior as current code
  </behavior>
  <action>
    1. In validation.rs:
       a. Add `jti_enforcement: EnforcementMode` field to `ValidationConfig`, replacing the boolean `enforce_jti`.
          Import `EnforcementMode` from `crate::policy::config`.
       b. Update `ValidationConfig::from_env()`:
          - Remove `UNIX_OIDC_DISABLE_JTI_CHECK` env var parsing
          - Set `jti_enforcement: EnforcementMode::Warn` as default (v1.0 behavior: JTI was checked but missing was warned, not rejected)
       c. In `TokenValidator::validate()`, replace the current JTI check block (lines 177-207) with enforcement-mode-gated logic:
          ```rust
          match jti_result {
              JtiCheckResult::Valid => {}
              JtiCheckResult::Replay => {
                  return Err(ValidationError::TokenReplay { jti: ... });
              }
              JtiCheckResult::Missing => {
                  match self.config.jti_enforcement {
                      EnforcementMode::Strict => {
                          tracing::warn!(check = "jti", mode = "strict", username = %claims.preferred_username,
                              "JTI missing - rejecting (strict mode)");
                          return Err(ValidationError::MissingJti);
                      }
                      EnforcementMode::Warn => {
                          tracing::warn!(check = "jti", mode = "warn", username = %claims.preferred_username,
                              "JTI missing - allowing with warning");
                      }
                      EnforcementMode::Disabled => {
                          tracing::debug!(check = "jti", mode = "disabled", "JTI check skipped");
                      }
                  }
              }
          }
          ```
          When `jti_enforcement` is `Disabled`, skip the entire JTI check (don't call `global_jti_cache().check_and_record()`).

    2. In auth.rs:
       a. In `authenticate_with_token()` and `authenticate_with_dpop()`: After loading `ValidationConfig::from_env()`, also try loading `PolicyConfig` to get security_modes. Thread the `jti_enforcement` from `PolicyConfig.effective_security_modes()` into `ValidationConfig.jti_enforcement`.
          ```rust
          let policy = PolicyConfig::from_env().ok();
          let security_modes = policy.as_ref()
              .map(|p| p.effective_security_modes())
              .unwrap_or_default();
          config.jti_enforcement = security_modes.jti_enforcement;
          ```
       b. For `authenticate_with_config()`: Accept a `ValidationConfig` directly (no change needed — caller controls the enforcement mode).

    3. Update existing tests in validation.rs:
       - All tests that set `enforce_jti: false` should use `jti_enforcement: EnforcementMode::Disabled`
       - All tests that set `enforce_jti: true` should use `jti_enforcement: EnforcementMode::Warn` (v1.0 default behavior)

    4. Add new tests:
       - `test_jti_strict_rejects_missing`: Create token without JTI, validate with strict mode -> MissingJti error
       - `test_jti_warn_allows_missing`: Create token without JTI, validate with warn mode -> Ok
       - `test_jti_disabled_skips_check`: Create token without JTI, validate with disabled mode -> Ok
       - `test_v1_default_behavior`: Validate that default EnforcementMode::Warn matches v1.0 behavior (warn on missing JTI, reject on replay)

    Note: DPoP enforcement mode wiring (dpop_required: strict/warn/disabled) affects auth.rs's `authenticate_with_dpop` function. For v1.0 defaults, `dpop_required: strict` is the current behavior (tokens with cnf.jkt MUST provide DPoP proof). The actual gating logic should check security_modes.dpop_required before requiring the proof. However, full DPoP enforcement mode wiring is better done when the DPoP nonce issuance lands in Phase 7. For now, thread the value but only implement the JTI enforcement gate. Add a TODO comment for DPoP enforcement mode in Phase 7.
