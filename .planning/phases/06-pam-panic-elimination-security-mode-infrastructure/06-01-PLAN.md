<objective>
Migrate all std::sync lock primitives to parking_lot and eliminate all production .unwrap()/.expect() panic sites in pam-unix-oidc.

Purpose: A PAM module panic can lock users out of their system. parking_lot eliminates PoisonError (no .unwrap() needed on lock acquisition), and the remaining panic sites (getrandom, reqwest builder, SystemTime) are converted to error propagation.

Output: All 32+ lock acquisition sites use parking_lot (no .unwrap()), 3 production .expect() calls replaced with Result propagation, JTI cache constant corrected to 100k.
</objective>

       - Remove ALL `.unwrap()` calls on `.read()` and `.write()` — parking_lot returns guards directly, no Result.
       - This is purely mechanical: `.read().unwrap()` becomes `.read()`, `.write().unwrap()` becomes `.write()`.

    3. In jti_cache.rs: Change `MAX_ENTRIES_BEFORE_CLEANUP` from `10000` to `100_000` (line 26). This resolves the SEC-07 code/documentation discrepancy. The dpop.rs `MAX_JTI_CACHE_ENTRIES` is already `100_000` — verify it stays consistent.

    4. In dpop.rs `validate_dpop_proof()` (line 280-283): Replace the `SystemTime::now().duration_since(UNIX_EPOCH).expect(...)` with `.unwrap_or_default()` — same pattern already used in session.rs line 38. A system clock before 1970 is misconfigured, but the PAM module must not panic. Returning duration 0 will cause the proof age check to fail naturally (proof will appear very old), which is the correct security-conservative behavior.

    5. In dpop.rs `maybe_cleanup()`: The double-lock pattern on lines 98-105 acquires `last_cleanup.read()` then `entries.write()` then `last_cleanup.write()`. With parking_lot this works fine (no poisoning), but simplify the write path: after `entries.retain()`, update `last_cleanup` within the same scope to avoid releasing and re-acquiring.

    Do NOT touch `lib.rs` `ENV_MUTEX` (std::sync::Mutex in test code) — this will be handled in plan 03 with the deny lint.
