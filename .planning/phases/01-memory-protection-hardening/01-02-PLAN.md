Purpose: Prevents token leakage via Debug/Display/logs and hardens the daemon process against memory inspection.
Output: Token fields wrapped in Secret<String>, process hardening at startup, mlock status in `status` output.

  <action>
    1. Add process hardening function to unix-oidc-agent/src/crypto/protected_key.rs (or a new `unix-oidc-agent/src/security.rs` module -- executor's discretion):
       - `pub fn disable_core_dumps()`:
         - `#[cfg(target_os = "linux")]`: `libc::prctl(libc::PR_SET_DUMPABLE, 0)` -- log result at INFO
         - `#[cfg(target_os = "macos")]`: `libc::ptrace(libc::PT_DENY_ATTACH, 0, std::ptr::null_mut(), 0)` -- log result at INFO
         - `#[cfg(not(any(target_os = "linux", target_os = "macos")))]`: log warning "Core dump disabling not supported on this platform"
       - Best-effort: log failures as WARN, never panic or fatal

    2. Change AgentState.access_token from `Option<String>` to `Option<Secret<String>>` in socket.rs:
       - Add `use secrecy::{Secret, ExposeSecret};`
       - In handle_request GetProof arm: change `let token = t.clone()` to `let token = t.expose_secret().clone()` -- this is the ONLY place raw token value is needed (sending to SSH client)
       - In AgentState::is_logged_in(): `self.access_token.is_some()` still works
       - In perform_token_refresh(): `state_write.access_token = Some(Secret::new(new_token))`
       - In extract_username_from_token calls: pass `token.expose_secret()`

    3. Update main.rs token handling:
       - In run_login(): after extracting access_token string from JSON, wrap: `let access_token = Secret::new(access_token.to_string())`
         - For storage.store(): `storage.store(KEY_ACCESS_TOKEN, access_token.expose_secret().as_bytes())?;`
         - For metadata JSON: refresh_token and client_secret fields need expose_secret for JSON serialization
       - In run_refresh(): same pattern -- wrap new tokens in Secret immediately after extraction
       - In load_agent_state(): wrap loaded access_token: `access_token.map(Secret::new)`
       - In run_logout/run_reset: these use storage.delete() on key names, no token values involved -- no changes needed

    4. Token metadata serialization: The metadata JSON object contains refresh_token and client_secret as serde_json::Value (from the IdP response). These are stored as JSON, not as Secret<String> fields directly. The main concern is:
       - In-memory `client_secret` variable in run_login: wrap in Secret if used beyond the spawn_blocking boundary
       - The metadata JSON value itself: the refresh_token and client_secret are serialized from serde_json::Value, not from String fields. The risk is the in-memory String variables. Wrap the local variables where they are held as Strings.

    5. Add mlock probe + prctl to daemon startup in run_serve():
       - Before `load_agent_state()`, call `disable_core_dumps()`
       - Call `mlock_probe()` and store the result
       - Log: `info!("Memory protection: mlock {}", status_description)`
       - Pass MlockStatus to AgentState (add a field) for status reporting

    6. Update Status response to include mlock state:
       - Add `mlock_status: Option<String>` field to AgentState
       - In the Status handler, include it in output
       - In run_status() CLI output: print "Memory protection: mlock active" or "Memory protection: mlock unavailable (reason)"
