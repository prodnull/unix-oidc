Wire DPoP nonce issuance into the PAM authentication conversation flow.

Purpose: Implements the two-round PAM keyboard-interactive conversation where round 1 delivers a server-generated nonce (DPOP_NONCE:<value>) and round 2 collects the nonce-bound DPoP proof. Output: Updated PAM authenticate() entry point that generates nonces, delivers them via conversation, and validates the nonce-bound proof. Includes a human verification checkpoint for the full flow.

  <files>pam-unix-oidc/src/lib.rs, pam-unix-oidc/src/auth.rs</files>
  <action>
Modify the PAM `authenticate()` function in `lib.rs` to support DPoP authentication with server-issued nonces. The current flow only uses `authenticate_with_token()`. The new flow adds a DPoP path that:

1. **Determine DPoP mode**: After rate-limit check and before token collection, load policy via `PolicyConfig::from_env().ok()` and check `effective_security_modes().dpop_required`. If not `Disabled`, proceed with DPoP nonce flow.

2. **Nonce generation and issuance** (before collecting proof):
   - Call `generate_dpop_nonce()` from security::nonce_cache.
   - Call `global_nonce_cache().issue(&nonce)` to register it in the cache.
   - Deliver nonce via PAM conversation round 1:
     ```rust
     let nonce_prompt = format!("DPOP_NONCE:{}", nonce);
     // Use PROMPT_ECHO_ON — client reads the prefix, agent extracts nonce,
     // responds with empty string or ack. This is how two-round keyboard-interactive works.
     if let Err(_) = pamh.conv(Some(&nonce_prompt), PamMsgStyle::PROMPT_ECHO_ON) {
         // Nonce delivery failed — fall back to non-DPoP path
         tracing::warn!("DPoP nonce delivery via PAM conversation failed, falling back to token-only auth");
     }
     ```

3. **Collect DPoP proof** (round 2):
   - Use `PROMPT_ECHO_OFF` for the proof (it's a secret):
     ```rust
     let dpop_proof = match pamh.conv(Some("DPOP_PROOF: "), PamMsgStyle::PROMPT_ECHO_OFF) {
         Ok(Some(p)) => Some(p.to_string_lossy().to_string()),
         _ => None,
     };
     ```

4. **Collect token** (round 3 or existing path):
   - Keep the existing `get_auth_token()` for the OIDC token itself.
   - Or combine: the client may send token + proof in a single response. Design for the two-value case where token comes from authtok/env and proof from conversation.

5. **Call authenticate_with_dpop()**:
   - Construct `DPoPAuthConfig` with `require_nonce: true`, `expected_nonce: None` (cache-backed mode), `target_host` from `gethostname`.
   - Call `authenticate_with_dpop(token, dpop_proof.as_deref(), &dpop_config)`.
   - The nonce validation happens inside `authenticate_with_dpop()` (wired in Plan 01) using `DPoPProofResult.nonce` from `validate_dpop_proof()`.

6. **Fallback behavior**:
   - If `dpop_required` is `Warn` or `Disabled` and no proof is provided, fall back to `authenticate_with_token()`.
   - If `dpop_required` is `Strict` and no proof is provided, return AUTH_ERR.

7. **Error handling**: All paths must be panic-free. Use `.map_err()` and early returns with PamError codes. Log all failures via AuditEvent.

8. **Important PAM conversation note**: The current `get_auth_token()` already uses `pamh.conv()` for token collection. The nonce delivery round must happen BEFORE the token/proof collection. Restructure the authenticate() function to:
   a. Rate limit check
   b. Load policy, determine DPoP mode
   c. If DPoP mode active: generate nonce, issue to cache, deliver via conv round 1
   d. Collect token (existing path via get_auth_token)
   e. Collect DPoP proof (new conv round 2 if DPoP mode)
   f. Authenticate (authenticate_with_dpop or authenticate_with_token based on mode)

All code must comply with deny(clippy::unwrap_used, clippy::expect_used). No panics.
