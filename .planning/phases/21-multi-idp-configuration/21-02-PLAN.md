<objective>
Wire multi-issuer routing into the authentication flow so the PAM module dispatches tokens to the correct per-issuer config, JWKS provider, and DPoP enforcement mode.

Purpose: MIDP-06 (issuer routing) and MIDP-07 (per-issuer JWKS caching) require the auth flow to extract `iss` from the token before validation, look up the matching IssuerConfig, and dispatch to the correct ValidationConfig and JwksProvider. The JTI cache must be scoped per-issuer to prevent cross-issuer replay false positives.

Output: `extract_iss_for_routing()` and `authenticate_multi_issuer()` in auth.rs; issuer-scoped JTI cache keys in jti_cache.rs; PAM entry points in lib.rs wired to the new dispatch; unit tests for routing, unknown issuer rejection, per-issuer DPoP enforcement, and JTI scoping.
</objective>

  <action>
1. Add `UnknownIssuer(String)` variant to `AuthError` enum in auth.rs.

2. Add `extract_iss_for_routing(token: &str) -> Result<String, AuthError>`:
   - Use `TokenClaims::from_token(token)` to decode payload without signature verification.
   - Return `claims.iss` normalized (trim trailing slash).
   - On decode failure, return `AuthError::Config(...)`.

3. Add `authenticate_multi_issuer()` function following the research Pattern 4 dispatch flow:
   ```rust
   pub fn authenticate_multi_issuer(
       token: &str,
       dpop_proof: Option<&str>,
       dpop_config: &DPoPAuthConfig,
       policy: &PolicyConfig,
       jwks_registry: &IssuerJwksRegistry,
   ) -> Result<AuthResult, AuthError>
   ```
   Implementation steps:
   a. Call `extract_iss_for_routing(token)` to get issuer URL.
   b. Call `policy.issuer_by_url(&iss)` — if `None`, return `AuthError::UnknownIssuer(iss)`.
   c. Build `ValidationConfig` from `IssuerConfig`: issuer_url as `issuer`, issuer `client_id`, JTI enforcement from `policy.effective_security_modes()`, clock skew from `policy.timeouts`.
   d. Get JWKS provider via `jwks_registry.get_or_init(&issuer_config.issuer_url, policy.cache.jwks_cache_ttl_secs, policy.timeouts.http_timeout_secs)`.
   e. Construct `TokenValidator::with_jwks_provider(validation_config, jwks_provider)` (or `new_insecure_for_testing` under test-mode feature).
   f. Validate token: `validator.validate(token)`.
   g. **Per-issuer DPoP enforcement** (MIDP-02): Use `issuer_config.dpop_enforcement` instead of global `SecurityModes.dpop_required`. Apply same DPoP proof validation + nonce consumption logic from `authenticate_with_dpop()` but with per-issuer enforcement mode.
   h. **Per-issuer claim mapping** (MIDP-03): Run `check_collision_safety(&issuer_config.claim_mapping)` then `UsernameMapper::from_config(&issuer_config.claim_mapping)` and `mapper.map(&claims)`.
   i. Build and return `AuthResult`.

4. **JTI cache scoping** (MIDP-07 anti-pattern avoidance): In `authenticate_multi_issuer()`, when calling `global_jti_cache().check_and_record()`, pass a scoped JTI:
   ```rust
   let scoped_jti = claims.jti.as_deref().map(|jti| format!("{}:{}", iss, jti));
   global_jti_cache().check_and_record(scoped_jti.as_deref(), &username, ttl);
   ```
   This is a caller-side change in auth.rs — does NOT require modifying `JtiCache` internals.

5. Keep the existing `authenticate_with_dpop()` and `authenticate_with_token()` functions unchanged for backward compatibility. The PAM entry point (Task 2) will choose which path to call based on whether `policy.issuers` is populated.

6. Write unit tests. For test-mode tests, create a helper that builds a minimal JWT with a specific `iss` claim (base64-encode `{"iss":"...","sub":"test","aud":"unix-oidc","exp":9999999999}` with a dummy header and signature). For DPoP per-issuer tests, verify that:
   - A token from issuer with `dpop_enforcement: Disabled` and no DPoP proof succeeds.
   - A token from issuer with `dpop_enforcement: Strict` and no DPoP proof returns DPoP-related error.

Anti-patterns to avoid:
- Do NOT use `PAM_USER` for routing. Routing key is ALWAYS `iss` from the token.
- Do NOT call `check_collision_safety` on a merged global config. Call per `IssuerConfig.claim_mapping`.
- Do NOT modify the JtiCache struct itself. The issuer scoping is done by prefixing the key at the call site.
