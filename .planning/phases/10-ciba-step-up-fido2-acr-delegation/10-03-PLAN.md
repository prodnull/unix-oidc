Wire CIBA step-up flow end-to-end: agent daemon handles StepUp IPC by initiating CIBA backchannel auth and spawning an async poll loop; PAM sudo.rs routes Push/Fido2 step-up methods to the agent via IPC.

After this plan, `sudo` commands with `step_up_method = "push"` or `"fido2"` trigger a CIBA push notification to the user's phone or FIDO2 authenticator.

Output: Working CIBA step-up flow from PAM sudo -> agent IPC -> IdP CIBA -> user approval -> PAM success/failure.

  <files>unix-oidc-agent/src/daemon/socket.rs, unix-oidc-agent/src/daemon/mod.rs</files>
  <behavior>
    - StepUp IPC dispatches to handle_step_up() which sends CIBA backchannel auth request and returns StepUpPending
    - A pending step-up is stored in a HashMap keyed by correlation_id with the Tokio JoinHandle
    - StepUpResult IPC checks the pending step-up: if task complete returns StepUpComplete or StepUpTimedOut, if still running returns StepUpPending
    - The CIBA poll loop sleeps for interval seconds between polls, adds 5s on SlowDown, returns error on AccessDenied/ExpiredToken
    - The poll loop respects timeout_secs: returns Timeout after deadline
    - When method is "fido2", acr_values=ACR_PHR is sent in the backchannel request; ACR validation runs on the returned token
    - When method is "push", no acr_values sent; no ACR validation on returned token
    - Concurrent StepUp for same username is rejected with an error (guard against double-initiation)
  </behavior>
  <action>
In `unix-oidc-agent/src/daemon/socket.rs`:

1. Add a `pending_step_ups: HashMap<String, PendingStepUp>` field to AgentState:
   ```rust
   pub struct PendingStepUp {
       pub handle: tokio::task::JoinHandle<StepUpOutcome>,
       pub username: String,
       pub expires_at: tokio::time::Instant,
   }

   pub enum StepUpOutcome {
       Complete { acr: Option<String>, session_id: String },
       TimedOut { reason: String, user_message: String },
   }
   ```

2. Add `handle_step_up()` function:
   - Check no pending step-up exists for this username (reject with error if one is active)
   - Read OIDC discovery via JwksProvider or by fetching discovery (may need to add issuer config to AgentState or pass config)
   - Construct CibaClient from discovery, client_id, client_secret
   - Build binding_message via `build_binding_message(&command, &hostname)`
   - Determine acr_values: if method == "fido2" then Some(ACR_PHR), else None
   - Send CIBA backchannel auth request (async reqwest POST to backchannel_endpoint with params)
   - Parse response as BackchannelAuthResponse
   - Generate correlation_id via uuid
   - Spawn async poll loop as Tokio task (see below)
   - Store PendingStepUp in HashMap keyed by correlation_id
   - Return AgentResponse::step_up_pending(correlation_id, auth_response.expires_in, auth_response.interval)

3. Implement async CIBA poll loop (runs as spawned Tokio task):
   ```rust
   async fn poll_ciba(
       http: reqwest::Client,
       token_endpoint: String,
       params: Vec<(String, String)>,  // owned copies of token poll params
       mut interval: Duration,
       timeout: Duration,
       acr_required: Option<String>,
   ) -> StepUpOutcome
   ```
   - Loop: sleep(interval), POST token_endpoint with params
   - On 200 OK: parse CibaTokenResponse. If acr_required is Some, validate_acr on id_token's acr claim. Return Complete.
   - On error response: parse error field. Match:
     - authorization_pending -> continue
     - slow_down -> interval += 5s, continue
     - access_denied -> return TimedOut("denied", "Step-up request was denied")
     - expired_token -> return TimedOut("expired", "Approval window expired")
     - other -> return TimedOut("error", format)
   - On timeout (tokio::time::timeout wrapping the loop): return TimedOut("timeout", "Step-up approval timed out")

4. Add `handle_step_up_result()` function:
   - Look up correlation_id in pending_step_ups HashMap
   - If not found: return error("Unknown step-up correlation ID", "STEP_UP_NOT_FOUND")
   - If task is finished (.is_finished()): remove from HashMap, await the JoinHandle, return StepUpComplete or StepUpTimedOut based on StepUpOutcome
   - If task still running: return StepUpPending with remaining time

5. Wire dispatch in `handle_connection()`:
   - `AgentRequest::StepUp { .. }` -> `handle_step_up()`
   - `AgentRequest::StepUpResult { .. }` -> `handle_step_up_result()`

6. In `unix-oidc-agent/src/daemon/mod.rs`, ensure any new pub items are exported.

**Configuration threading:** The agent needs issuer, client_id, client_secret to construct CIBA requests. These are likely already available from the login flow configuration. If not in AgentState, add fields for them or read from stored config. The agent already knows the issuer (it fetched tokens from it during login). Thread the config through handle_step_up.

**HTTP client:** Reuse the async reqwest::Client from the agent's existing infrastructure (or create one per step-up session with appropriate timeout). Do NOT create per-poll-request clients.

**Security:** ACR validation for fido2 method is HARD-FAIL — if the IdP returns a token without the required ACR, the step-up MUST fail. This is a security invariant (not configurable via EnforcementMode).

Note: The PAM module deny(clippy::unwrap_used, clippy::expect_used) applies to pam-unix-oidc. The agent crate may have different lint settings — follow whatever is currently in place for unix-oidc-agent.

In `pam-unix-oidc/src/sudo.rs`, extend the `perform_step_up()` function to handle `StepUpMethod::Push` and `StepUpMethod::Fido2`:

1. Currently `perform_step_up()` only checks `requirements.allowed_methods.contains(&StepUpMethod::DeviceFlow)`. Add branches for Push and Fido2 that route through agent IPC:
   - Determine method string: "push" for StepUpMethod::Push, "fido2" for StepUpMethod::Fido2
   - Get hostname via `gethostname()` (or `hostname::get()` if available, otherwise `libc::gethostname`)
   - Construct StepUp IPC message: `AgentRequest::StepUp { username, command, hostname, method, timeout_secs: requirements.timeout }`
   - Send to agent via existing Unix socket IPC (use the same pattern as SessionClosed in Phase 9 — std::os::unix::net::UnixStream with timeout)
   - Parse response: expect StepUpPending with correlation_id
   - Enter PAM-side poll loop: sleep poll_interval_secs, send StepUpResult{correlation_id}, check response
     - StepUpPending -> continue (still waiting)
     - StepUpComplete -> extract acr, validate if needed, return Ok
     - StepUpTimedOut -> display user_message via StepUpDisplay, return SudoError::Timeout or SudoError::Denied
   - Total PAM-side loop bounded by requirements.timeout (do not exceed LoginGraceTime)
   - Each IPC call is short/bounded (2s timeout per call, same as SessionClosed pattern)

2. In `pam-unix-oidc/src/policy/config.rs`, verify `SudoConfig.challenge_timeout` defaults to 120 (update if currently different). This satisfies STP-07 (configurable step-up timeout, default 120s). The field already exists — just confirm the default.

3. Add `SudoError::StepUp(String)` variant for CIBA-specific step-up errors.

4. Update `log_step_up_initiated()` to include the step-up method in the audit log.

**Key constraint:** The PAM-side poll loop does SHORT IPC calls (2s each). The CIBA polling (5s+ intervals for 120s) happens entirely in the agent daemon's async Tokio task. The PAM thread just checks status. This avoids the LoginGraceTime race (Pitfall 3 from research).

Note: The agent IPC socket path is resolved via `UNIX_OIDC_AGENT_SOCKET` env or `XDG_RUNTIME_DIR` fallback, same as the Phase 9 SessionClosed pattern in `pam-unix-oidc/src/session/mod.rs` `notify_agent_session_closed()`.
