# Keycloak DPoP + Device Authorization Grant: Reference Implementation

## Overview

Brief statement: Keycloak 26.0+ is the only IdP that supports DPoP-bound token issuance
on the Device Authorization Grant flow. This document describes the configuration required
and how to verify it works.

## What This Proves

- Keycloak issues access tokens with `cnf.jkt` claim (RFC 9449) when a DPoP proof is
  presented during token acquisition
- The PAM module validates the DPoP proof against the token's `cnf.jkt` thumbprint
- Tokens acquired without a DPoP proof are rejected when `dpop.bound.access.tokens` is enabled

## Quick Start: Run the Reference Stack

Reference `docker-compose.e2e.yaml` as the canonical runnable example:

```bash
docker compose -f docker-compose.e2e.yaml up -d
# Wait for Keycloak health check
curl -sf http://localhost:9000/health/ready
# Run DPoP binding verification
bash test/tests/test_dpop_binding.sh
```

## Required Keycloak Realm Configuration

Extract the following from the actual `unix-oidc-realm.json` fixture and present inline:

1. **Client configuration** — Extract from the realm JSON:
   - `directAccessGrantsEnabled` (for ROPC test path)
   - `publicClient` vs confidential
   - OAuth Device Authorization Grant enabled (the attribute/setting that enables device flow)
   - DPoP bound access tokens setting (`dpop.bound.access.tokens` or equivalent attribute)

2. **Realm settings** — Any realm-level settings required for DPoP

3. **Token settings** — Token lifetimes, signature algorithm (RS256)

IMPORTANT (D-06): Every setting name, attribute key, and Keycloak admin path must be verified
against the actual realm JSON fixture AND Keycloak 26.x documentation. Do NOT use setting names
from training data that may be incorrect for Keycloak 26.5.5. Read the realm JSON to find the
exact attribute names Keycloak uses.

## What to Verify

Describe the three verification checks (matching what test_dpop_binding.sh does):

1. Token request WITH DPoP proof returns 200 and access token contains `cnf.jkt`
2. `cnf.jkt` value matches the computed JWK thumbprint (RFC 7638) of the DPoP proof's public key
3. Token request WITHOUT DPoP proof is rejected (400) when binding is required

Include the curl + jq one-liner to inspect cnf.jkt in a token.

## Keycloak Admin UI Path (for manual setup)

For operators not using realm import, document the Keycloak Admin Console path:
- Realm Settings -> Client -> [client name] -> Settings tab
- The specific toggles/settings for DPoP and Device Flow

Again, verify these paths against Keycloak 26.x documentation.

## Standards References

- RFC 9449 — Demonstrating Proof of Possession (DPoP)
- RFC 7638 — JSON Web Key (JWK) Thumbprint
- RFC 8628 — OAuth 2.0 Device Authorization Grant
