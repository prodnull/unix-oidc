<objective>
Implement RFC 7662 token introspection as an opt-in validation step in the PAM authentication flow, with moka-based result caching and configurable fail-open/fail-closed behavior.

Purpose: Introspection enables near-real-time token revocation detection. When an employee's account is disabled at the IdP, their next auth attempt fails within the cache TTL.
Output: Working introspection client with caching, wired into the authenticate() flow.

  <behavior>
    - introspect_token() with enabled=false returns Ok(true) without making HTTP call
    - introspect_token() with enabled=true and active=true response returns Ok(true)
    - introspect_token() with enabled=true and active=false response returns Ok(false)
    - introspect_token() with unreachable endpoint + enforcement=Warn returns Ok(true) and emits IntrospectionFailed audit event
    - introspect_token() with unreachable endpoint + enforcement=Strict returns Err
    - Cache hit: second call within TTL returns cached result without HTTP call
    - Cache key uses JTI claim, not raw token value (security: no bearer credential as map key)
    - Cache key falls back to sha256(token[..32]) when JTI absent
    - Cache TTL capped at min(configured_ttl, token_exp - now)
    - Introspection uses HTTP POST with application/x-www-form-urlencoded per RFC 7662 section 2.1
    - Client authenticates to introspection endpoint via Basic Auth with client_id/client_secret
    - Adversarial: malformed JSON response from endpoint treated as introspection failure
    - Adversarial: endpoint returns 401 (invalid credentials) treated as failure with appropriate log
  </behavior>
  <action>
1. Create pam-unix-oidc/src/oidc/introspection.rs:

   Error type:
   ```rust
   #[derive(Debug, thiserror::Error)]
   pub enum IntrospectionError {
       #[error("HTTP request failed: {0}")]
       Http(String),
       #[error("Response parse failed: {0}")]
       Parse(String),
       #[error("Introspection endpoint not configured")]
       NotConfigured,
       #[error("Token is inactive (revoked or expired at IdP)")]
       TokenInactive,
   }
   ```

2. IntrospectionCache:
   - Wraps moka::sync::Cache<String, bool> (key: JTI or sha256 digest, value: active boolean).
   - Constructor: new(max_capacity: u64, default_ttl_secs: u64).
   - Method: get_or_insert(cache_key: &str, token_exp: i64, introspect_fn: impl FnOnce() -> Result<bool, IntrospectionError>) -> Result<bool, IntrospectionError>.
   - On cache miss: call introspect_fn, compute effective_ttl = min(default_ttl, max(0, token_exp - now)), insert with per-entry TTL via Cache::policy().set_time_to_live() — actually moka sync Cache TTL is per-cache not per-entry. So instead, use the default TTL set at cache construction and rely on the global TTL being the configured value. The token_exp cap is a best-effort protection: if token_exp < now + cache_ttl, log a WARN that cache may outlive token. In practice the 60s default is short enough this rarely matters.
   - Use process-global singleton via once_cell::sync::Lazy (same pattern as DPoPNonceCache).

3. Public API function:
   ```rust
   pub fn introspect_token(
       config: &IntrospectionConfig,
       token: &str,
       token_jti: Option<&str>,
       token_exp: i64,
       client_id: &str,
       client_secret: Option<&str>,
       session_id: Option<&str>,
       username: Option<&str>,
   ) -> Result<bool, IntrospectionError>
   ```
   - If !config.enabled: return Ok(true) immediately.
   - If config.endpoint is None: return Err(NotConfigured) — caller handles per enforcement mode.
   - Compute cache_key: use token_jti if Some, else sha256(token.as_bytes()[..min(32, token.len())]) hex-encoded.
   - Call cache.get_or_insert() with closure that does the HTTP POST.
   - HTTP POST: reqwest::blocking::Client with 5s timeout, form body [("token", token), ("token_type_hint", "access_token")], Basic Auth with client_id + optional client_secret. Parse response JSON, extract "active" boolean (RFC 7662 section 2.2: REQUIRED field).
   - On HTTP or parse error: emit IntrospectionFailed audit event and return Err.

4. Add `pub mod introspection;` to pam-unix-oidc/src/oidc/mod.rs.

IMPORTANT: The reqwest blocking client must be built once and reused. Use once_cell::sync::Lazy for the global client. This avoids per-call TLS handshake overhead.

1. In authenticate(), after successful token validation (in the Ok(result) arm, after username match check, before the putenv and SUCCESS return):

   - Load IntrospectionConfig from PolicyConfig::from_env().
   - If introspection is enabled, call introspect_token() with:
     - config: &policy.introspection
     - token: &token
     - token_jti: result.token_jti.as_deref()
     - token_exp: result.token_auth_time (or extract exp from token claims — use the exp that auth.rs already extracts)
     - client_id: from OIDC_CLIENT_ID env var (already used elsewhere in this file)
     - client_secret: None (for now — introspection endpoint auth config is Claude's discretion; use Basic Auth with just client_id for now, add client_secret to IntrospectionConfig if needed)
     - session_id: Some(&result.session_id)
     - username: Some(&result.username)

   - Handle result based on enforcement mode:
     - Ok(true): token is active, continue to SUCCESS.
     - Ok(false): token is inactive/revoked.
       - Strict: record failure, emit audit event, return AUTH_ERR.
       - Warn: log warning, continue to SUCCESS.
       - Disabled: unreachable (introspection disabled).
     - Err(IntrospectionError::NotConfigured):
       - Strict: log error ("introspection enabled but no endpoint configured"), return SERVICE_ERR.
       - Warn: log warning, continue.
     - Err(other):
       - Strict: record failure, return AUTH_ERR.
       - Warn: log warning (fail-open), continue.

2. Add IntrospectionError to the AuthError enum or handle it inline in lib.rs (prefer inline handling since introspection is a post-validation check, not part of the core auth flow).

3. Add `client_secret` field to IntrospectionConfig if needed for the Basic Auth pattern. Per RFC 7662 section 2.1, the introspection endpoint MUST require client authentication. The client_secret should come from IntrospectionConfig (operator configures it in policy.yaml alongside the endpoint).

IMPORTANT: The introspection check runs AFTER token signature verification, issuer/audience checks, and DPoP validation. It is an additional active-status check, not a replacement for cryptographic validation.
