pub trait SecureStorage: Send + Sync {
    fn store(&self, key: &str, value: &[u8]) -> Result<(), StorageError>;
    fn retrieve(&self, key: &str) -> Result<Vec<u8>, StorageError>;
    fn delete(&self, key: &str) -> Result<(), StorageError>;
    fn exists(&self, key: &str) -> bool;
}

#[derive(Debug, Error)]
pub enum StorageError {
    #[error("Key not found: {0}")]
    NotFound(String),
    #[error("Access denied")]
    AccessDenied,
    #[error("Storage backend error: {0}")]
    Backend(String),
    #[error("Encryption error: {0}")]
    Encryption(String),
    #[error("IO error: {0}")]
    Io(#[from] std::io::Error),
}

    **Step 1 — Cargo.toml feature fix (CRITICAL prerequisite):**
    Change `keyring = "3"` to:
    ```toml
    keyring = { version = "3", features = ["sync-secret-service", "linux-native", "apple-native"] }
    ```
    This is the confirmed bug from research: without features, keyring uses a mock store on Linux.

    **Step 2 — CI fix:**
    Add `libdbus-1-dev` to every `apt-get install` line in `.github/workflows/ci.yml` that installs `libssl-dev` (7 locations). The `sync-secret-service` feature requires `dbus-1` at compile time.

    **Step 3 — StorageError extension:**
    Add a `Migration(String)` variant to `StorageError` in `mod.rs` for migration failures. Add `pub mod router;` and re-export `StorageRouter`, `BackendKind`, `MigrationStatus`.

    **Step 4 — router.rs implementation:**
    Create `unix-oidc-agent/src/storage/router.rs` with:

    `BackendKind` enum: `SecretService`, `KeyutilsUser`, `MacOsKeychain`, `File` — each with `display_name() -> &'static str` returning the user-facing strings from CONTEXT.md ("keyring (Secret Service)", "keyring (keyutils @u)", "keyring (macOS Keychain)", "file (fallback)").

    `MigrationStatus` enum: `Migrated(usize)`, `NotMigrated`, `NotApplicable` — with `display_name()` returning "migrated", "not migrated", "n/a".

    `StorageRouter` struct with fields:
    - `backend: Box<dyn SecureStorage>` (the active backend)
    - `kind: BackendKind` (which backend was selected)
    - `migration_status: MigrationStatus` (set after migration attempt)

    `StorageRouter::detect() -> Result<Self, StorageError>`:
    1. Check `UNIX_OIDC_STORAGE_BACKEND` env var. If set, call `detect_forced()` which probes ONLY the requested backend. If probe fails, return `Err(StorageError::Backend("forced backend '...' failed probe: ..."))` — per user decision, do NOT fall through.
    2. On Linux: probe Secret Service first via `set_default_credential_builder(keyring::secret_service::default_credential_builder())` then write/read/delete sentinel. If OK, return `BackendKind::SecretService` with a `KeyringStorage`.
    3. On Linux: probe keyutils via `set_default_credential_builder(keyring::keyutils::default_credential_builder())` then write/read/delete sentinel. If OK, return `BackendKind::KeyutilsUser`.
    4. On macOS: probe macOS Keychain (the default for apple-native). If OK, return `BackendKind::MacOsKeychain`.
    5. Fall through to `FileStorage::new()` with `BackendKind::File`.
    Log INFO for selected backend, INFO for each failed probe explaining why.

    Probe function: Use sentinel key `unix-oidc-probe` with service `unix-oidc-agent`. Write `b"probe-ok"` (base64 encoded via KeyringStorage), read back, delete. Use `Entry::new()` directly for probe — do NOT use `KeyringStorage` struct for probing because the probe needs to test with the specific credential builder set. Actually, since `KeyringStorage` uses `Entry::new()` which picks up the global default, set the builder THEN use `KeyringStorage` for the probe cycle.

    `impl SecureStorage for StorageRouter`: delegate all methods to `self.backend`.

    **Step 5 — Unit tests using keyring::mock:**
    Use `keyring::mock::default_credential_builder()` via `set_default_credential_builder()` in tests. The mock backend always succeeds, simulating a working keyring. For failure tests, use a custom approach (e.g., set DBUS_SESSION_BUS_ADDRESS to empty to force Secret Service probe failure on non-macOS, or test the `detect_forced` path with an invalid backend name).

    **Important design notes from research:**
    - Call `set_default_credential_builder()` BEFORE any `Entry::new()` call
    - Probe sentinel value must be non-empty (keyutils rejects empty payloads — Pitfall 4)
    - The `linux-native` backend's persistent keyring link provides @u-equivalent persistence (Research Finding 2) — the spike aspect is that this code running in CI will empirically confirm the keyutils path works
