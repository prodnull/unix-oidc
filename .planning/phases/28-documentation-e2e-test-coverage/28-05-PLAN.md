Find the Keycloak realm configuration file for the CIBA test realm. It may be in `test/docker/keycloak/` as a JSON export, or defined in the docker-compose.ciba-integration.yaml via environment variables or volume mounts.

**If a ciba-realm.json or similar export file exists:** Update it to add ACR-to-LoA mapping entries. Specifically:

Add to the Keycloak realm JSON under `"attributes"` (or the Keycloak Level of Assurance policy configuration):
- ACR value `"phr"` maps to LoA level 3 (phishing-resistant, simulates FIDO2 AAL3)
- ACR value `"phrh"` maps to LoA level 4 (hardware-bound, simulates hardware FIDO2)
- ACR value `"mfa"` maps to LoA level 2 (standard MFA)

In Keycloak 26.x, LoA configuration in realm JSON looks like:
```json
"attributes": {
  "acr.loa.map": "{\"phr\":\"3\",\"phrh\":\"4\",\"mfa\":\"2\",\"default\":\"1\"}"
}
```

Also ensure the `unix-oidc-ciba` client has `"attributes"` with `"backchannel.authentication.request.signing.alg": "none"` (or appropriate) and that the CIBA authentication flow for the realm allows the auto-approval path via the Admin API.

**If no ciba-realm.json exists (realm configured entirely via Admin API at startup):** Create `test/docker/keycloak/ciba-realm-acr-patch.sh` instead — a script that patches the existing realm via Keycloak Admin API to add LoA mapping:

```bash
#!/usr/bin/env bash
# Patch ciba-test realm to add FIDO2 ACR simulation via LoA mapping
KEYCLOAK_URL="${KEYCLOAK_URL:-http://localhost:8080}"
ADMIN_TOKEN=$(curl -sf -X POST "${KEYCLOAK_URL}/realms/master/protocol/openid-connect/token" \
  -d "grant_type=password&client_id=admin-cli&username=admin&password=admin" | jq -r '.access_token')

curl -sf -X PUT "${KEYCLOAK_URL}/admin/realms/ciba-test" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"attributes": {"acr.loa.map": "{\"phr\":\"3\",\"phrh\":\"4\",\"mfa\":\"2\"}"}}'

echo "Realm ciba-test patched with FIDO2 ACR LoA mapping"
```

Update the docker-compose.ciba-integration.yaml to call this patch script during startup (or add it to the Keycloak `KC_SPI_*` environment variables if realm export is not available).

Document in a comment: "Real FIDO2 hardware is not required — Keycloak LoA levels 3/4 are assigned to phr/phrh ACR values to simulate phishing-resistant authentication for CI. The test validates ACR claim presence and value, not the underlying authenticator."
