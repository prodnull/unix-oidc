<objective>
Add HMAC-based tamper-evidence chain to audit events and ship a verification utility.

Purpose: Security auditors need assurance that audit logs have not been tampered with. An HMAC chain (each event includes HMAC of previous hash + current payload) provides verifiable integrity — any deletion or modification breaks the chain at the tampered event. This satisfies OBS-06.
Output: HMAC chain fields on all audit events, `unix-oidc-audit-verify` subcommand for chain validation.
</objective>

  <action>
    In `pam-unix-oidc/src/audit.rs`:

    1. Add a module-level static for the chain state:
       ```rust
       use hmac::{Hmac, Mac};
       use sha2::Sha256;
       type HmacSha256 = Hmac<Sha256>;

       static CHAIN_STATE: Lazy<Mutex<ChainState>> = Lazy::new(|| Mutex::new(ChainState::new()));

       struct ChainState {
           hmac_key: Option<Vec<u8>>,
           prev_hash: String,  // "genesis" for first event
       }
       ```

    2. In `ChainState::new()`, read `UNIX_OIDC_AUDIT_HMAC_KEY` env var. If absent or empty, set `hmac_key = None` and log WARNING "Audit HMAC chain disabled — UNIX_OIDC_AUDIT_HMAC_KEY not set. Set this env var to enable tamper-evident audit logging."

    3. Create `ChainState::compute_chain(&mut self, event_json: &str) -> Option<(String, String)>`:
       - If `hmac_key` is None, return None (chain disabled)
       - Compute HMAC-SHA256 of `format!("{}:{}", self.prev_hash, event_json)` using the key
       - Return `Some((prev_hash, chain_hash))` where chain_hash is the hex-encoded HMAC
       - Update `self.prev_hash = chain_hash.clone()`

    4. Modify `log()` to:
       a. First serialize the event to JSON (existing behavior)
       b. If chain is enabled, compute chain fields and create an enriched JSON:
          ```rust
          #[derive(Serialize)]
          struct ChainedEvent<'a> {
              #[serde(flatten)]
              event_json: &'a serde_json::Value,  // original event as Value
              prev_hash: String,
              chain_hash: String,
          }
          ```
       c. Use the chained JSON for syslog, file, and stderr output
       d. If chain is disabled, use the original JSON (no chain fields)

    5. Add `hmac` and `sha2` to pam-unix-oidc/Cargo.toml dependencies. These are already transitive deps via `ring`/`jsonwebtoken` but should be explicit.

    **Threading safety:** The `Mutex<ChainState>` ensures chain ordering is correct even under concurrent PAM calls in the same sshd process. In the forked-sshd model, each fork gets its own chain (which is correct — each fork processes one session).

    **Security note:** The HMAC key should be a high-entropy secret (at least 32 bytes). Document this in the log-retention guide. The key must be the same across all processes writing to the same log file for chain verification to work. In the forked-sshd model, the parent sshd sets the env var and all forks inherit it.

<task type="auto" tdd="true">

  <files>pam-unix-oidc/src/bin/audit_verify.rs, pam-unix-oidc/Cargo.toml</files>
  <behavior>
    - Test 1: Given a valid chain log file, the verifier reports "Chain verified: N events, 0 breaks"
    - Test 2: Given a log file with a modified event, the verifier reports the line number and event of the first break
    - Test 3: Given a log file with a deleted event, the verifier reports the break at the gap
    - Test 4 (negative): Given a log file without chain fields (chain was disabled), the verifier reports "No chain fields found — tamper-evidence was not enabled"
    - Test 5: Given an empty file, the verifier reports "No events found"
  </behavior>
  <action>
    Create `pam-unix-oidc/src/bin/audit_verify.rs`:

    1. A binary that reads a log file line by line, parses each line as JSON, extracts `prev_hash` and `chain_hash` fields.

    2. For each event:
       a. Strip `prev_hash` and `chain_hash` from the JSON to get the bare event payload
       b. Recompute HMAC-SHA256 of `format!("{}:{}", prev_hash, bare_event_json)` using the provided key
       c. Compare recomputed hash to `chain_hash`
       d. If mismatch, report the line number and event type

    3. CLI interface:
       ```
       unix-oidc-audit-verify --key <hmac-key> --file <log-file>
       unix-oidc-audit-verify --key-file <path-to-key-file> --file <log-file>
       ```
       The key can be provided as a hex string via `--key` or read from a file via `--key-file`.

    4. Output format:
       ```
       Verifying audit chain: /var/log/unix-oidc-audit.log
       Events processed: 1234
       Chain status: VALID (no breaks detected)
       ```
       Or on failure:
       ```
       Verifying audit chain: /var/log/unix-oidc-audit.log
       Events processed: 567
       CHAIN BREAK at line 234:
         Event: SSH_LOGIN_SUCCESS
         Expected prev_hash: abc123...
         Found prev_hash: def456...
         The event at line 234 or the event at line 233 may have been tampered with.
       Chain status: INVALID (1 break detected)
       ```

    5. Exit code: 0 for valid chain, 1 for broken chain, 2 for usage error.

    6. Add to `pam-unix-oidc/Cargo.toml`:
       ```toml
       [[bin]]
       name = "unix-oidc-audit-verify"
       path = "src/bin/audit_verify.rs"
       ```

    7. Use `clap` for argument parsing (already a workspace dependency via unix-oidc-agent).

    **Security consideration:** The verification key must match the key used to generate the chain. The binary should never log the key or include it in error messages.

    **Testing:** Create integration tests that:
    - Write a temp log file with chained events, verify it passes
    - Modify one event in the file, verify it reports the break
    - Delete one event, verify it reports the break
    - Write events without chain fields, verify it reports "not enabled"
  </action>
