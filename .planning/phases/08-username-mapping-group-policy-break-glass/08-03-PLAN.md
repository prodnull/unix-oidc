  <action>
Replace the advisory warn-only block in `auth.rs` with a hard-fail call to `check_collision_safety()`.

Current code (approximately lines 101-109):
```rust
let mapper = policy_opt.as_ref().map(|policy| {
    let warnings = crate::identity::collision::validate_collision_safety(&policy.identity);
    for w in &warnings {
        tracing::warn!(collision_warning = %w, "Username mapping collision safety warning");
    }
    UsernameMapper::from_config(&policy.identity)
});
```

Replace with:
```rust
let mapper = policy_opt
    .as_ref()
    .map(|policy| {
        crate::identity::collision::check_collision_safety(&policy.identity)
            .map_err(|e| AuthError::Config(e.to_string()))?;
        UsernameMapper::from_config(&policy.identity)
    })
    .transpose()?;
```

Note the use of `.transpose()?` to propagate the `Result` out of the `Option::map` closure. `mapper` type changes from `Option<Result<UsernameMapper, _>>` to `Option<UsernameMapper>` (error is propagated immediately). Update downstream match arms in `authenticate_with_token` and `authenticate_with_dpop` accordingly — they currently match on `Some(Ok(ref m))` and `Some(Err(e))`; after the transpose these become `Some(ref m)` and `None`.

The same construction pattern appears in both `authenticate_with_token` (the standalone function) and `authenticate_with_config` / `authenticate_with_dpop`. Apply the hard-fail to all three entry points that construct the mapper from policy.

Add `use crate::identity::collision::check_collision_safety;` (or use the full path) — whichever is consistent with existing import style in the file.

Security rationale: IDN-03 is a HARD-FAIL invariant per CONTEXT.md — same class as signature verification. No configurable bypass. The error message surfaced to the PAM caller will read "Configuration error: Non-injective username transform pipeline detected..." which is operator-facing at config load time and safe to reveal transform details to.
