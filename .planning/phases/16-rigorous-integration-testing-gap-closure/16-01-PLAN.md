# Plan 16-01: CIBA-Enabled Keycloak Realm + CI Integration

**Phase:** 16 — Rigorous Integration Testing (Gap Closure)
**Requirements:** INT-01, INT-04
**Depends on:** Phases 10, 14 (CIBA implementation + bug fixes)

## Goal

Build the CI test infrastructure for CIBA backchannel authentication and ACR validation against a live Keycloak instance. This is the foundation for validating that Phase 10's CIBA step-up and ACR delegation actually work end-to-end.

## Tasks

### Task 1: CIBA-Enabled Keycloak Test Realm

Create `test/fixtures/keycloak/ciba-test-realm.json`:

1. New realm `ciba-test` with CIBA poll mode enabled on the client
2. Client `unix-oidc-ciba` — confidential, CIBA grant enabled, DPoP bound
   - `oidc.ciba.grant.enabled: true`
   - `dpop.bound.access.tokens: true`
   - `backchannel.token.delivery.mode: poll`
3. ACR LoA mapping configured via authentication flows:
   - LoA 1: password-only (default browser flow)
   - LoA 2: password + OTP (maps to ACR `urn:mace:incommon:iap:silver` or similar)
   - ACR mapper protocol mapper on client
4. Test users: `cibauser/cibapass` (with OTP configured if needed for ACR tests)
5. Service account enabled on client for Admin API auto-approval

### Task 2: Docker Compose for CIBA Integration

Create `docker-compose.ciba-integration.yaml`:

1. Keycloak 26.2 with `--features=token-exchange,admin-fine-grained-authz,dpop,ciba`
2. Import both existing realm JSONs + new `ciba-test-realm.json`
3. Health check on port 9000
4. Expose port 8080

### Task 3: CIBA Integration Test Script

Create `test/tests/test_ciba_integration.sh`:

1. Wait for Keycloak ready
2. Obtain admin access token via client credentials
3. Initiate CIBA backchannel auth request (POST to backchannel_authentication_endpoint)
4. Auto-approve via Keycloak Admin REST API (simulate user approval)
5. Poll token endpoint with CIBA grant type until success
6. Validate response contains access_token with ACR claim
7. Validate ACR claim value against expected LoA mapping (INT-04)
8. Negative test: verify expired CIBA request returns appropriate error
9. Negative test: verify denied CIBA request returns access_denied

### Task 4: CI Workflow Job

Add `ciba-integration` job to `.github/workflows/ci.yml`:

1. Depends on `check`
2. Starts `docker-compose.ciba-integration.yaml`
3. Runs `test/tests/test_ciba_integration.sh`
4. Collects logs on failure
5. Stops environment on cleanup

## Success Criteria

- [ ] Keycloak starts with CIBA feature enabled and realm imported
- [ ] CIBA backchannel auth request returns `auth_req_id`
- [ ] Admin API auto-approval triggers token availability
- [ ] Token poll returns access_token with ACR claim present
- [ ] ACR value matches configured LoA mapping
- [ ] CI job runs the test on push/PR

## Security Considerations

- Test client secret is hardcoded in realm JSON (test-only, never used in production)
- CIBA auto-approval via Admin API is test-only behavior; production CIBA requires real user interaction
- ACR validation test confirms the IdP's LoA mapping is honored, not bypassed
